# Claude Code 记忆系统深度解读：AI 是如何「记住」你的？

> 基于 Claude Code 源代码的记忆工程逆向分析。面向对 AI 感兴趣但不一定会编程的读者。

---

## 引言：AI 的「金鱼记忆」困境

你有没有这样的体验？每次和 AI 助手开始新对话，它都像失忆了一样，完全不记得你上次说了什么、你是做什么工作的、你喜欢怎样的回答风格。你不得不一遍又一遍地重复同样的信息。

这是因为大语言模型（LLM）本质上是「无状态」的——每次对话都是一次全新的开始。它没有真正的「记忆」，只有当前对话窗口里的上下文。

Anthropic 的 Claude Code（一款基于命令行的 AI 编程助手）为这个问题设计了一套精巧的**文件式记忆系统**。更有趣的是，它还实现了一个叫做 **autoDream**（自动做梦）的功能——让 AI 在你不知不觉中，像人类睡眠时做梦一样，自动整理和巩固记忆。

本文将带你深入解读这套系统的核心原理。

---

## 一、整体架构：三层记忆体系

Claude Code 的记忆系统由三个核心模块组成，就像人脑的不同记忆区域：

### 1. memdir —— 记忆的「档案馆」

这是记忆系统的基础设施层。它负责：

- **存储**：所有记忆以 Markdown 文件的形式存在磁盘上（路径类似 `~/.claude/projects/<项目名>/memory/`）
- **索引**：每个记忆目录有一个 `MEMORY.md` 入口文件，充当目录索引（类似图书馆的书目卡片），限制在 200 行 / 25KB 以内
- **分类**：记忆被严格归为四种类型
- **检索**：通过一个轻量 AI 模型（Sonnet）来动态匹配与当前问题相关的记忆

### 2. extractMemories —— 实时「记笔记」的速记员

在你和 AI 每一轮对话结束后，系统会在后台悄悄启动一个「分身」（forked agent），回顾刚才的对话内容，提取值得长期保存的信息并写入记忆文件。你不需要任何操作，它全自动运行。

### 3. autoDream —— 定期「做梦」整理记忆

这是最有创意的部分——它模拟人类睡眠中的记忆巩固过程。每隔一段时间，系统会自动启动一次「梦境整理」，回顾近期多次对话的记录，合并、修正、删除过时的记忆。

三者的关系可以用一个通俗的比喻来理解：

| 模块            | 比喻         | 职责                         |
| --------------- | ------------ | ---------------------------- |
| memdir          | 🏛️ 档案室     | 提供存储、索引与检索基础设施 |
| extractMemories | 📝 随堂速记员 | 每轮对话后提取要点           |
| autoDream       | 🌙 睡眠整理师 | 定期整合、去重、修正、清理   |

## 二、记忆的四种类型：AI 记住什么？

系统对记忆内容进行了严格的分类，而不是什么都存。这四种类型是：

### 🧑 用户记忆（user）
关于「你是谁」的信息。比如你的职业角色、技术背景、工作习惯。

> 例：「用户是数据科学家，目前关注可观测性和日志系统」
> 例：「用户有十年 Go 经验，但第一次接触 React 前端」

一个资深工程师和一个编程初学者，需要截然不同的沟通方式——这类记忆让 AI 能做到这一点。

### 💬 反馈记忆（feedback）
你对 AI 工作方式的纠正或认可。这是最重要的记忆类型之一。

> 例：「不要在测试中 mock 数据库——上季度因为 mock 测试通过但生产环境迁移失败而出了事故」
> 例：「不要在每次回答末尾做总结，用户说他自己会看 diff」

关键设计：系统不仅记录纠正（「不要这样做」），也记录认可（「对，就这样做」）。如果只记住犯过的错误，AI 会变得过度谨慎，丢掉已经验证有效的做法。每条反馈都要附上**原因**（Why），这样 AI 在面对边界情况时能做出灵活判断，而不是生搬硬套。

### 📋 项目记忆（project）
关于当前项目的动态信息——那些从代码和 git 历史中看不出来的东西。

> 例：「代码合并冻结从 2026-03-05 开始，因为移动端团队要切发布分支」
> 例：「替换旧认证中间件，驱动力是法务合规要求，不是技术债务」

注意一个精妙的细节：系统要求将相对日期（如「周四」「下周」）转换为绝对日期。因为记忆可能在数周后被读取，到那时「周四」早已无从判断是哪个周四了。

### 📌 参考记忆（reference）
外部系统的指引。

> 例：「Pipeline 相关 bug 在 Linear 项目 "INGEST" 中追踪」
> 例：「grafana.internal/d/api-latency 是值班延迟看板」

### 同样重要的：不记什么

系统明确规定了**不应保存**的内容：

- ❌ 代码模式、架构、文件路径——这些可以直接从当前代码库中获取
- ❌ Git 历史——`git log` 才是权威来源
- ❌ 调试方案——修复已经在代码里了
- ❌ 临时任务状态——当前对话结束就没用了

即使用户明确说「帮我记住这个 PR 列表」，系统也被指示要追问：「其中什么是**意外的**或**非显而易见的**？那部分才值得留下。」

## 三、记忆的书写与检索

### 写入：每条记忆都是一个文件

每条记忆是一个独立的 Markdown 文件，带有结构化的「头部信息」（frontmatter）：

```markdown
---
name: 数据库测试策略
description: 集成测试必须使用真实数据库，禁止 mock
type: feedback
---

集成测试必须连接真实数据库，不使用 mock。

**Why:** 上季度因为 mock 与生产环境行为不一致，
导致迁移失败未被测试发现。

**How to apply:** 编写涉及数据层的测试时，
始终配置连接到测试数据库实例。
```

同时还要在 `MEMORY.md` 索引文件中加入一行指引：
```
- [数据库测试策略](feedback_testing.md) — 禁止 mock，必须连真实数据库
```

### 检索：AI 帮 AI 选记忆

当用户发起新请求时，系统不会把所有记忆一股脑塞进上下文，而是：

1. **扫描**所有记忆文件的头部信息（文件名、描述、类型、最后修改时间）
2. 将这个「目录清单」连同用户的当前问题，发送给一个轻量 AI 模型（Sonnet）
3. 该模型从中**最多选出 5 条**最相关的记忆
4. 只有被选中的记忆才被读取全文并注入到主对话上下文中

这就像「请图书馆员根据你的问题，从书架上挑出最相关的 5 本书」——而不是把整个图书馆搬过来。

### 鲜度感知：记忆会「褪色」

系统会计算每条记忆距今的天数，超过 1 天的记忆会附上警告：

> 「这条记忆已有 47 天。记忆是某个时间点的快照，不是实时状态——关于代码行为或文件位置的描述可能已过时。在当作事实引用前请先验证。」

这避免了 AI 把过时信息当作真理来推荐。

## 四、extractMemories：无声的速记员

每次你和 AI 完成一轮完整的问答后，`extractMemories` 模块会在后台自动运行。它的工作流程是：

```text
1. 用户提问 → AI 回答
2. （后台）启动提取记忆子代理
3. 分析最近的对话内容
4. 扫描已有记忆目录（防重复提取）
5. 更新已有记忆文件 或 创建新记忆
6. 在聊天界面隐式/显式提示「已保存 N 条记忆」
```

几个关键设计：

- **互斥机制**：如果主 AI 在对话中已经主动写了记忆（比如用户说「帮我记住…」），后台速记员就跳过这一轮，避免重复
- **权限最小化**：速记员只能读取文件和在记忆目录内写入，不能执行写入命令、不能修改代码
- **预算控制**：最多 5 个来回，防止陷入无用的验证循环
- **效率优先**：先并行读取所有可能需要更新的文件，再并行写入——两步内完成

## 五、autoDream：AI 的「睡眠整理」

这是整个记忆系统中最富创意的部分。

### 灵感来源

对于人类来说，睡眠不仅仅是休息——大脑在睡眠期间会进行「记忆巩固」（memory consolidation）：把白天获取的碎片化信息重新整合，强化重要的、丢弃无用的、修正矛盾的。神经科学研究表明，梦境可能就是这个过程的一种表象。

Claude Code 的 `autoDream` 就是这个过程的软件模拟。

### 触发条件：三道门槛

autoDream 不是随时运行的。它有精心设计的「门控」（gate）机制，按照计算成本从低到高依次检查：

#### 第 1 关：时间门槛
距离上次整理是否已过足够时间？（默认 ≥ 24 小时）

这只需要检查一个锁文件的修改时间——极低成本。

#### 第 2 关：对话量门槛
在这段时间内是否积累了足够多的对话？（默认 ≥ 5 次会话）

扫描会话文件目录，排除当前正在进行的会话。

#### 第 3 关：锁定机制
是否有其他进程已经在执行整理？

通过写入进程 ID（PID）到锁文件来实现互斥。即使进程崩溃（PID 不再运行），系统也能检测到并回收锁定。锁定超时时间为 1 小时，防止僵死状态。

只有三道门槛全部通过，「做梦」才会真正启动。

### 梦境的四个阶段

一旦启动，autoDream 会按照严格的四阶段流程执行：

#### 阶段一：定向（Orient）
- 查看记忆目录中现有的所有文件
- 阅读 `MEMORY.md` 索引
- 浏览已有的主题文件，了解当前记忆的全貌

#### 阶段二：采集新信号（Gather）
在近期的对话记录中寻找值得记住的新信息。按优先级：
1. 日志文件（如果有的话）
2. 已存在但可能已经过时的记忆——与代码库当前状态矛盾的
3. 对话原文——用精确关键词搜索，而不是通读全部

#### 阶段三：整合（Consolidate）
这是核心工作：
- **合并**：把新信息融入已有的主题文件，而非创建近似重复
- **时间校正**：将「昨天」「上周」之类的模糊表述转换为确切日期
- **修正**：如果今天发现某条旧记忆与事实矛盾，直接修改源头

#### 阶段四：修剪与索引（Prune & Index）
- 更新 `MEMORY.md` 索引，保持在 200 行 / 25KB 限制内
- 删除指向已过时或被取代记忆的链接
- 压缩过长的索引条目——如果概要超过 200 字符，说明它承载了本应在主题文件中的内容
- 解决矛盾——如果两个文件观点冲突，修正错误的那个

### 安全与容错

autoDream 的设计体现了工程上的严谨：

- **只读 Shell**：做梦期间只能执行 `ls`、`find`、`grep`、`cat` 等只读命令，任何写入文件系统的 Shell 命令都会被拒绝
- **写入范围限定**：只能编辑记忆目录内的文件，不能触碰项目代码
- **崩溃回滚**：如果做梦过程意外失败，锁文件的时间戳会被回退到执行前的状态，确保下次还能正常触发
- **用户可中止**：用户可以随时通过后台任务面板终止做梦进程
- **扫描节流**：即使时间门槛已经过了，如果 10 分钟内刚刚扫描过会话目录，就不会重复扫描
- **进度可见**：做梦过程的每一步都会更新到任务面板中，用户能看到它在做什么、修改了哪些文件

## 六、记忆的安全防线

记忆系统涉及文件读写，安全至关重要。代码中可以看到多层防护：

### 路径安全
- **禁止路径穿越**：拦截 `../`、URL 编码的 `%2e%2e%2f`、Unicode 全角字符伪装等各种攻击手段
- **符号链接防护**：写入前检查真实路径，防止通过符号链接写到记忆目录之外（比如 `~/.ssh/`）
- **禁止危险起点**：拒绝根目录 `/`、Windows 盘根 `C:\`、UNC 网络路径

### 权限控制
- 对来自项目配置文件（`.claude/settings.json`）的路径覆盖持不信任态度——恶意仓库可能试图通过这个渠道获取敏感目录的写权限
- 后台代理只有最小权限：能读任意文件，但只能写入记忆目录

### 敏感数据
- 团队共享记忆中禁止保存 API 密钥、用户凭据等敏感信息

## 七、记忆的「可信度」哲学

代码中最耐人寻味的，是它对「记忆可信度」的深思熟虑。系统反复强调一个核心理念：

> **「记忆说 X 存在」不等于「X 现在存在」。**

具体体现为：

1. **验证优先**：如果记忆中提到某个函数名，用它之前先 grep 确认函数还在
2. **现实优先**：如果记忆与当前代码矛盾，以当前代码为准，并更新记忆
3. **标注时效**：超过 1 天的记忆自动附上「可能已过时」的提醒
4. **尊重遗忘**：用户说「忽略记忆」时，AI 应彻底忽略，不要出现「虽然记忆中说了 X，但我将忽略」这种画蛇添足

这反映了一种成熟的设计哲学：**记忆是辅助工具，不是权威来源。它应该让 AI 更有上下文地工作，但永远服从于当前的客观事实。**

## 八、总结：从「金鱼记忆」到「有阅历的伙伴」

Claude Code 的记忆系统通过三个协作模块，实现了一个从无到有、持续进化的记忆生命周期：

```
对话进行中
    │
    ├── extractMemories：从对话中实时提取要点 → 写入记忆文件
    │
    ├── memdir：存储、索引、按需检索相关记忆 → 注入下一次对话
    │
    └── autoDream：定期启动梦境整理 → 合并、修正、清理记忆
```

它的设计充满了对现实世界复杂性的尊重：

- 记忆不是越多越好，而是要**精准、结构化、可维护**
- 旧记忆不是不可更改的圣经，而是**可验证、可修正、可删除**的快照
- 自动化不意味着失控，而是有**门控、权限、回滚**的严格工程保障
- AI 不是被动的记录者，而是一个会**主动整理和反思**的知识管理者

## 九、延伸思考：文件记忆的天花板在哪里？

Claude Code 的记忆系统在 AI 编程助手中堪称标杆。但作为技术解读的尾声，值得追问一个问题：**Markdown 文件能撑起一个 AI 的「长期大脑」吗？**

有三个结构性的张力值得留意：

**记忆越多，维护越贵。** 每次检索要扫描所有文件头部信息，每次做梦要通读索引和相关文件。200 行的 `MEMORY.md` 索引是硬上限——记忆丰富到这个天花板后，压缩索引会损失检索精度。这是一个**文件越多 → 成本越高 → 准确率越低**的循环。对于一个项目的辅助记忆绰绰有余，但承载数年的持续认知，它会被自身的重量压垮。

**文件之间没有「关系」。** 假设 AI 分别记住了「Alice 负责 Aurora 项目」「Aurora 从 MySQL 迁移到 PostgreSQL」「Alice 擅长数据库优化」——在人脑中这三条信息自动编织成网络，能瞬间推导出「Alice 最适合处理 Aurora 数据库问题」。但在 Markdown 里它们散落在三个文件中，没有显式链接，AI 只能靠自然语言理解去「猜」关联。这种沿关系链条的**多跳推理**，恰恰是平面文件结构的盲区。

**矛盾处理丢失了演化轨迹。** autoDream 发现过时记忆后直接修改或删除——简单有效，但「我们三个月前用 MySQL，上周迁到了 PostgreSQL」这条演化时间线就丢失了。未来有人问「技术栈经历过哪些变化？」时，答案已经不在了。

这些并不是 Claude Code 的「缺陷」——在它的使用场景下，这些都是合理的工程权衡。但它们指向一个更深层的问题：**要让 AI 拥有真正的长期记忆，底层数据结构可能需要从「文件」进化到「图谱」**——一个由实体（节点）和关系（边）编织成的网络，天然支持关系遍历、矛盾演化追踪和跨事件模式提取。

如果你对这个方向感兴趣，可以看看 [**Anda Brain**](https://github.com/ldclabs/anda-brain)——一个借鉴人脑大脑机制、用知识图谱替代 Markdown 文件来实现 AI 记忆巩固的开源项目。它的核心理念是：**消耗的词元不应该蒸发，而应该结晶为可复用、可迁移、不绑定特定模型的结构化认知资产。**

---

从「每次对话都是陌生人」到「一个记得你、了解你、持续成长的合作伙伴」——Claude Code 的记忆系统让 AI 朝着这个方向迈出了扎实而优雅的一步。而其中 autoDream 的「做梦整理」隐喻，更是工程美学与认知科学碰撞出的一个精彩注脚：**即使是机器，要想「记得好」，也需要偶尔「停下来想一想」。**
